# Designing a Dependency Injection system for Swift in 2026

The strongest convergent recommendation across Apple's sample code, Airbnb, Lyft, Uber, Point-Free, and the most influential Swift writers (Seemann, Sundell, van der Lee, Long, Williams/Celis, Jabrayilov, Wals, Fatbobman) is a hybrid: **interface/implementation module split enforced by SPM, concrete witness structs as service types, key-path keyed registration with required default values, and `@TaskLocal`-scoped overrides for tests and previews**. This pattern matches what `swift-dependencies` ships, what Apple's `@Environment`/`@Entry` system encodes, and what large modular shops have independently arrived at — yet none of the existing libraries hit every requirement for a from-scratch system aimed at modular Swift 6 codebases. Compile-time graph validation, available only in Needle and SafeDI today, is the missing piece worth building. The decision space is no longer "container vs. no container" but rather "how much of Dagger's compile-time graph correctness can we absorb into Swift's type system without paying SwiftSyntax build-time costs everywhere?" This report maps the design space, names the battle-tested primitives, and distills the open questions a from-scratch implementation must resolve.

## Theoretical foundations matter more than they appear

Three concepts are routinely conflated and must be kept separate. **Inversion of Control** is a generic principle about *who calls whom* — frameworks, callbacks, the Hollywood principle. **The Dependency Inversion Principle** is the SOLID "D" and concerns the *direction of source-code dependency edges*: high-level modules and low-level modules both depend on abstractions. **Dependency Injection** is one *technique* that achieves both, by passing collaborators rather than constructing them. You can have DIP without DI (depending on protocols and resolving statically through generics) and DI without DIP (constructor-injecting a concrete type). For a Swift system, this matters because the language's type system already gives you DIP for free through `protocol`/`some`/`any` — your DI mechanism only needs to handle the *injection* concern.

The injection-style hierarchy is settled: **constructor injection is the default**, property injection is the exception for genuinely optional or framework-instantiated dependencies, and method injection is reserved for per-call context (a `Clock` passed into one operation). This is Mark Seemann's recommendation in *Dependency Injection: Principles, Practices, and Patterns* (2nd ed., Manning 2019), and it is echoed by Sundell, van der Lee, Wals, and the entire Point-Free corpus. Property injection's chief pathology — leaving an object in a half-constructed state with implicit-unwrap or optional fields — maps directly to the SwiftUI `@EnvironmentObject` crash mode that Apple itself moved away from with `@Observable`.

The **Service Locator pattern is rightly considered an anti-pattern** outside the Composition Root. Seemann's argument, refined across three blog posts (2010, 2014, 2015), is that a class calling `Locator.resolve(Foo.self)` hides preconditions: the public API does not advertise that `Foo` must be registered, so misconfiguration surfaces at runtime, not compile time. It also exposes an unbounded interface — the locator promises "any service" — making test substitution near-impossible without supplying everything. **Swinject, Resolver, and Factory's lookup style are service locators by Seemann's definition; `swift-dependencies` is technically also one, but its `@TaskLocal`-based scoping makes it a *scoped* locator rather than a global one, which closes most of the holes**. The acceptability boundary is that calls *inside the Composition Root* are not service location — they are composition.

The **Composition Root concept** — Seemann's term for the unique location, as close to the entry point as possible, where modules are wired — translates directly to Swift as `@main App.init`, `AppDelegate.application(_:didFinishLaunchingWithOptions:)`, or a dedicated `AppComposition.swift`. Each test method is its own Composition Root, which is why per-test mock registration is not a service-locator violation. Crucially, **libraries and frameworks must not have a Composition Root** — they expose DI-friendly types and let consumers compose. This rule alone resolves most of the "where does the container live in a modular SPM project?" debates: nowhere except the App target.

**Scopes and lifetimes** divide cleanly into singleton (one per container), transient (new every resolution), scoped/per-graph (one per logical unit of work), and weak. Seemann's rule of thumb — pick the *shortest* lifetime that satisfies the requirements, and never let a longer-lived object hold a shorter-lived one without a `Provider<T>` indirection — is what every well-known DI framework encodes as a constraint, and it remains the right rule for a Swift design. **Cyclic dependencies are almost always a design smell**: two classes "needing each other" usually means a third concept is missing or an SRP violation. The mechanical fixes (extract a mediator, replace a direct reference with `Provider<T>`/lazy/closure, use property injection on one side) are well-trodden but should be exceptional escape hatches, not first-line tools.

## Compile-time DI in Swift: a tiered field

Compile-time DI is the primary target because it eliminates whole classes of bugs (missing registrations, wrong scope) at build time rather than at runtime or test time. The Swift ecosystem has six serious entrants, of which only **two — Needle and SafeDI — provide true graph-level compile-time correctness**.

**Needle (Uber)** is the canonical reference implementation, open-sourced May 2018. Architecture: each scope is a `Component<DependencyProtocol>` paired with a `DependencyProtocol` that declares what the scope receives from ancestors. The root is a `BootstrapComponent`. A separate `needle generate` CLI parses sources via SwiftSyntax, walks the component tree, and emits a single `NeedleGenerated.swift` whose `DependencyProvider` structs satisfy each protocol property by traversing parent components. **If a child requires `scoreStream` and no ancestor provides it, the build aborts with `Could not find a provider for (scoreStream: MyScoreStream) which was required by ScoreSheetDependency, along the DI branch of RootComponent->LoggedInComponent->ScoreSheetComponent.`** This is the gold-standard error message in Swift DI. Needle scales to multi-million-line codebases (Uber's Rider/Driver), supports hierarchical "generational" dependencies (an `AuthenticatedUser` only exists below `LoggedInComponent`), and compiles cleanly under Swift 6 strict concurrency, though its `Scope`/`Component` types are class-based and predate the modern `Sendable` conventions. Verified status as of late 2025: v0.25.1, in **maintenance mode** (last PR merged Q4 2025), but still building cleanly. The cost is steep: each scope needs a Component plus a DependencyProtocol; the CLI binary is awkward to integrate with SwiftPM build plugins; SwiftUI value-type ergonomics are mediocre because Components are reference types.

**SafeDI (Dan Federman, ex-Airbnb)** is the modern macro-driven equivalent. Latest release 1.5.4 (March 25, 2026); 2.0 in flight. Macros: `@Instantiable` opts a type into the graph, `@Instantiated` marks owned dependencies, `@Received` marks dependencies that come from ancestors (no protocol needed), `@Forwarded` marks runtime-derived values (an authenticated user) injected non-optionally at a subtree boundary. The build plugin reads source via SwiftSyntax, validates the full graph, and emits a single tree. Missing dependencies produce **inline IDE diagnostics with fix-its** — strictly better than Needle's CLI text errors. It also generates full-subtree mocks (`@Instantiable(generateMock: true)`), supports multi-module projects via `.safedi/configuration/include.csv`, and ships a prebuilt-binary plugin that dodges the SwiftSyntax compile-time tax. **For a greenfield project that needs Needle-class safety, SafeDI is the recommended choice today.**

**Cleanse (Square)** was the Swift port of Dagger 2's binding-DSL approach. Officially **archived June 13, 2024**; do not adopt. **Weaver (Scribd)** offered annotation-driven codegen via SourceKit; **archived** by Scribd. **Knit (Cash App)** is a hybrid: at runtime it uses Swinject, but Knit parses your `Assembly` source files at build time and generates type-safe `Resolver` extension methods plus a `DuplicateRegistrationDetector` Swinject behavior. Active (v2.1.3, Aug 2025); the **best migration path for codebases already on Swinject** that want compile-checked resolution at the edges, but not a true graph validator — cross-module duplicates can only be caught at runtime container assembly, and there is no hierarchical scoping by construction.

A "Carpenter" Swift DI library could not be located in this research; treat as unconfirmed. A "Bloomberg Knit" does not exist — the Knit project is from Cash App. These should be flagged before relying on any specific implementation reference.

The Swift 5.9+ macro system has unlocked a new wave of DI tooling. Beyond SafeDI, the notable entrants are **Crocodil** (`@DependencyEntry` macro registers a key path on a `Dependencies` namespace; closer to a service-locator shape than a graph DI but compile-validated), **`@DependencyClient`** in `swift-dependencies` (synthesizes a struct-of-closures interface with default unimplemented endpoints), and several smaller community macros (`Ryu0118/swift-dependencies-macro`, the experimental `SwiftDI` from Swift Forums). **No macro DI yet matches SafeDI's full-graph validation plus hierarchical scoping**; the rest are roughly "service-locator with compile-time key-path validation."

The compile-time tier list as of 2026:

| Approach | Mechanism | Modular fit | SwiftUI fit | Concurrency | Activity | Tier |
|---|---|---|---|---|---|---|
| SafeDI | Swift macros + SwiftPM plugin | Excellent | Excellent | Native Swift 6 | Very active | **A** |
| swift-dependencies + `@DependencyClient` | Macro + TaskLocal | Excellent | Excellent | Native | Very active | **A** |
| Needle | External CLI codegen | Excellent | Mediocre | Compiles clean | Maintenance | **B** |
| Knit + Swinject | CLI codegen on Swinject | Good | Good | OK | Active | **B+** |

## Pure-Swift compile-time techniques without any library

The most underrated insight in modern Swift DI is that **Swift's type system already gives you most of what a container provides**, if you use it correctly. Four pure-Swift techniques deserve to be the foundation of any from-scratch design.

**Protocol witnesses** — replacing a Swift `protocol` with a `struct` of closures — are the most transformative. Point-Free covered this in episodes #33–#36; the insight is that Swift compiles protocols to witness tables of function pointers anyway, so you can do the same by hand and pick up composability, multiple "conformances," generics-free call sites, and trivial mocking. A `struct UserClient { var fetch: @Sendable (UUID) async throws -> User; var save: @Sendable (User) async throws -> Void }` admits as many static instances (`live`, `mock`, `preview`, `unimplemented`) as you need, can be transformed (`pullback`, logging wrappers, retry wrappers) by ordinary value-level operations, and is `Sendable` for free when its closures are `@Sendable`. The cost is structural: forgetting a field is caught at construction, not at conformance, and the witness must be passed explicitly. **This is the strictly dominant style for shared/cross-actor services in Swift 6**, because protocol-based DI fights `Sendable` and PATs constantly.

**Manual Composition Root (Pure DI)** is what Seemann recommends as the default. Every dependency wired by hand at `@main`. It scales to small and medium apps, and it gives compile-time safety for the entire wiring graph including lifetimes — a guarantee no container can match. The Point-Free `isowords` open-source app, Apple's Backyard Birds and Scrumdinger samples, and Tuist's TMA reference projects all use Pure DI as the baseline. It breaks down only when feature-module count grows past ~50 and the App target becomes a 1,000-line wiring file — at which point you split by per-feature builders rather than reach for a container.

**Phantom types and parameterized modules** distinguish scopes and named instances at compile time without strings. `enum AppScope {}; enum UserScope {}; struct Container<Scope> { ... }` makes a `Container<UserScope>` unrepresentable from a `Container<AppScope>` context. Point-Free's `swift-tagged` (Episode #12) crystallizes the pattern. Useful for tagging two `URL`s as `PrimaryAPIURL` vs `MetricsURL` without a wrapper protocol.

**Key-path-based DI** is the SwiftUI/Apple convention: `KeyPath<Container, Service>` as the identity. SwiftUI's `@Environment(\.font)` and `swift-dependencies`' `@Dependency(\.uuid)` are the same pattern. Key paths give refactor safety, prevent typos at compile time, and admit a required-default-value design that makes "service not registered" a *type error* rather than a runtime miss. **This is the single most important design decision for a from-scratch system**: use `KeyPath<Values, Service>` as the key, never strings.

A fifth technique — **environment-passing (Reader-monad-style)** — threads an explicit `Env` struct through every layer. It is compile-safe by construction but ergonomically awkward in Swift without macro-level sugar. `swift-dependencies`' `DependencyValues` is spiritually a reader environment; the `@TaskLocal` underneath hides the threading.

## Runtime DI: Swinject, Factory, swift-dependencies, and the design trade

Runtime DI exists for ergonomics, not safety. **Swinject** is the classic IoC container — `Container.register`, `Resolver.resolve`, `ObjectScope` for lifetimes (`.transient`, `.graph`, `.container`, `.weak`). It is **maintained but not aggressively evolving** (last commit Sep 2025); known issues include non-thread-safe `Container` by default with `synchronize()` deadlock risks (Issue #266 — `NSLock` is non-reentrant) and a `Lazy<T>` factory that bypasses the synchronized wrapper (Issue #493). Storyboard support is increasingly irrelevant. Common criticisms: force-unwrap-heavy API, runtime crashes for unregistered dependencies, no compile-time safety.

**Factory (Michael Long)** is the modern non-TCA pragmatic pick. Latest line **Factory 2.5.x**, ~2.6k stars, very active. The architectural innovation: dependencies are **computed properties returning `Factory<T>` values** on a container, so forgetting to register is a *compile error* (missing property), not a runtime crash. Internally the resolution is still a dictionary keyed by `StaticString` + container, but the graph of *known* dependencies is fixed at compile time. Property wrappers — `@Injected`, `@LazyInjected`, `@WeakLazyInjected`, `@InjectedObject`, `@InjectedObservable` (iOS 17+) — give the SwiftUI ergonomics. Scopes: `.unique`, `.singleton`, `.cached`, `.shared`, `.graph`, custom. Factory 2.5 added Swift Testing parallelization support, strict concurrency under Swift 6, and a `FactoryTesting` library to keep test targets clean. **Resolver**, Long's predecessor, is **officially deprecated**; migrate.

**`swift-dependencies` (Point-Free)** is the most influential modern Swift DI library and deserves a deep read. The architecture mirrors `EnvironmentValues`: a `DependencyKey` protocol with `liveValue`/`testValue`/`previewValue`, a `DependencyValues` struct accessed via key path, and the `@Dependency(\.foo)` property wrapper. The novelty is the **`@TaskLocal` storage**: when you call `withDependencies { $0.api = .mock } operation: { ... }`, the library copies the current `DependencyValues`, applies your mutations, and binds them on a static `@TaskLocal` for the duration of the operation. **Child structured tasks (`async let`, `withTaskGroup.addTask`) inherit the binding automatically; detached/escaping closures lose it** (use `withEscapedDependencies` to capture and re-bind). This is the architectural innovation that distinguishes it from Swinject/Factory — overrides are *lexically scoped to a task tree* rather than globally registered, so two parallel test cases overriding `\.api` to different values do not race. Three modes per dependency — `liveValue` (production, lazy `static let`), `testValue` (unimplemented sentinels that fail-on-call via `Issue.record`/`XCTFail`), `previewValue` (deterministic safe defaults for `#Preview`) — give the test/preview/live trichotomy that no other Swift library provides.

The `@DependencyClient` macro (1.1, Nov 2023; refined through 2024–2025) generates the memberwise initializer, default unimplemented closures, and named-argument shim methods for a struct-of-closures dependency, dramatically reducing boilerplate. Recent releases added `@Dependency(_:as:)` for force-casting existential dependencies (1.8), SwiftUI Environment integration (1.7), and Swift Testing trait support (1.10). The library is `Sendable`-required by design and has a strong Swift 6 story.

The cons are real and acknowledged by Brandon Williams in interviews: **it is a runtime service locator under the hood**. The TaskLocal scoping closes most of the holes, but a missing `liveValue` is still a runtime resolution path; "session-based" dependencies (a `currentUser` only valid post-login) don't fit cleanly (Discussion #42); `withDependencies(from:)` uses `Mirror` reflection and is fragile if the parent has no `@Dependency` properties; `withTaskGroup`-body bindings are illegal (Discussion #195); and the model fits *single-entry-point* architectures (TCA reducers, SwiftUI bodies) far more cleanly than scattered MVVM callback chains.

The runtime DI verdict, opinionated: **`swift-dependencies` is the right pick for any project where TaskLocal scoping makes sense and where the "ambient injection" model fits — TCA, modern SwiftUI, server-side Swift. Factory is the right pick for codebases that want a more conventional container-of-keypaths model with first-class SwiftUI integration and no TaskLocal complexity. Swinject and Resolver should be migrated away from.**

## SwiftUI's native DI: powerful primitive, dangerous default

SwiftUI's `@Environment` plus `@Observable` is Apple's *implicit* DI system. The mechanics are key-path-keyed lookup against an `EnvironmentKey` protocol whose only requirement is a `defaultValue`. Xcode 16's **`@Entry` macro** (WWDC 2024) collapses the boilerplate to one line and is **backwards-compatible to iOS 13** because it expands to the same `EnvironmentKey` declaration. With iOS 17's `@Observable`, you can `.environment(myObservable)` and read it back as `@Environment(MyObservable.self)` without an explicit key — SwiftUI uses the type as the identity. Per-property re-rendering replaces the old `@Published` invalidate-everyone semantics, which is a substantial DI win because injected `@Observable` services no longer cause unnecessary re-renders.

The trade-offs are sharper than the API suggests. **Three failure modes recur in production code:**

First, the `defaultValue` requirement actively encourages *bogus* defaults — `fatalError()` placeholders or no-op live instances — that silently mask missing injections. The whole appeal of compile-time DI is making missing dependencies impossible; an `EnvironmentKey` whose default is a `LiveAPIClient()` quietly puts you on the network in previews and tests.

Second, `@Environment(MyObservable.self)` (the type-based form) **has no `defaultValue` and crashes at runtime** if no ancestor injected the value — the same failure mode as the old `@EnvironmentObject`. The mitigation is an optional declaration (`@Environment(AuthService.self) private var auth: AuthService?`), but most code skips it.

Third, **view-models cannot read `@Environment` directly**. This is Lucas van Dongen's chicken-and-egg critique: any architecture with view-models needs a separate DI mechanism for the model layer, so `@Environment` cannot be the *only* DI strategy. Apple's response, visible in Backyard Birds and Scrumdinger, is to dispense with view-models entirely and put state in `@Observable` models accessed via `@Environment` — but that is a top-to-bottom architectural commitment, not a DI choice.

**The community consensus is therefore narrow**: `@Environment` is excellent for ambient cross-cutting concerns (theme, locale, feature flags, an authenticated user, a shared `@Observable` store), but should not be the primary DI mechanism for app-wide services in non-trivial apps. Antoine van der Lee, Yang Xu (Fatbobman), Majid Jabrayilov, and Donny Wals all treat it as *a* DI tool; Lucas van Dongen and Brandon Williams treat it as *partial* DI; Apple itself never says "use `@Environment` for DI" — only that it is the SwiftUI data-flow primitive.

**Preview support is non-negotiable for any new DI design.** Previews run in a sandboxed process; views that reach a global singleton at init crash with opaque errors. The `XCODE_RUNNING_FOR_PREVIEWS == "1"` environment variable is the standard escape hatch but encourages branching live code. The right pattern is `swift-dependencies`' `previewValue` — a deterministic, side-effect-free implementation that runs by default in `#Preview`. Factory ships an analogous `.preview { }` modifier and `.context(.preview)`. Pure init injection plus the Xcode 16 `@Previewable` macro makes this trivial for small apps. **Any DI design that does not give a first-class preview-safe value will lead to widespread `XCODE_RUNNING_FOR_PREVIEWS` hacks and broken previews.**

## Swift 6 strict concurrency reshapes the DI design space

Five Swift 6 constraints rewrite the rules:

`Sendable` is now enforced for everything crossing an isolation boundary. Every dependency factory closure that runs from another isolation domain must be `@Sendable`; every value stored in a container that other tasks read must be `Sendable`. The protocol-witness style — a struct of `@Sendable` closures — is `Sendable` for free, which is one of the strongest reasons closure-of-structs has won over protocol-with-mock-class for shared services.

Global variables must be safe (SE-0412). The classic `static let shared = ...` singleton trips `Static property 'shared' is not concurrency-safe...` warnings. The escapes are `Sendable` conformance, `@MainActor` isolation, or `nonisolated(unsafe)` with manual discipline. **Any from-scratch DI design must avoid the single-shared-container model** unless that container is itself `Sendable` or actor-isolated.

Property-wrapper actor inference is gone (SE-0401). The old rule that `@StateObject` propagated `@MainActor` to its containing View no longer applies; instead the `View` protocol itself is `@MainActor` in Swift 6. This breaks code that called nonisolated init from `body` and explains many of the migration warnings on existing DI containers.

`@TaskLocal` is the **only Swift primitive today that gives DI the trifecta of thread-safe, scoped, and automatically inherited by child tasks**. SE-0311 (approved 2021, ships in Swift 5.5) is what makes `swift-dependencies`' `withDependencies` design possible. The caveat that TaskLocal does *not* propagate to GCD threads matters: any DI container layered on TaskLocal silently breaks when consumers use `DispatchQueue.global()`.

Swift Testing runs tests **in parallel in the same process** by default — XCTest used separate processes per file. **Any DI library based on global mutable state silently leaks state between tests under Swift Testing**. Point-Free's claim that `swift-dependencies` is the only library fully compatible with Swift Testing is, as of mid-2025, accurate; Factory 2.5 added explicit Swift Testing parallelization support, and Swift 6.1's `TestScoping` traits give a structured way for any library to participate. **A from-scratch design must build for parallel tests from day one.**

## Modular SPM architecture: the interface/implementation split is universal

The single most convergent finding across Airbnb (1.5M LOC, ~1,500 modules), Lyft (1.5M LOC, ~700 modules), Bumble (110+ modules), Spotify, Uber, monday.com, Back Market, Trendyol, and the Tuist TMA reference projects is the **interface module / implementation module split**. Every feature ships:

- A **`FeatureInterface`** package with protocols, value types, and factory signatures. Zero or minimal dependencies. Imported by feature consumers and the App.
- A **`FeatureImpl`** package with concrete classes, networking, persistence. Depends on `FeatureInterface`. Imported only by the App target.
- An optional **`FeatureMocks`** or **`FeatureTestSupport`** package depending only on `FeatureInterface`.

Feature modules **cannot see each other** — they communicate only through interface packages. This collapses the horizontal dependency graph from O(n²) to O(n), enables parallel SPM builds, makes interface-only packages compile in milliseconds, lets previews and mocks link against tiny surface areas, and turns "add a feature" into "add a pair of packages plus one line in App." Michael Bachand's 2021 Airbnb post, "Designing for Productivity in a Large-Scale iOS Application," is the canonical published statement: *"Feature modules are not visible to other feature modules; they share lightweight types via a sibling module type called a feature interface."* Tuist's TMA goes further with five target types (Source, Interface, Tests, Testing/mocks, Example app); Lyft enforces a **UI / Logic / Service** taxonomy with validators that reject UI imports from Logic targets.

**The "god container" anti-pattern** — a single `AppContainer` in `App.swift` that registers every dependency — fails predictably at scale: every feature must import the container module, transitive build dependencies explode, the container becomes a merge-conflict hotspot, tests drag the entire object graph, and missing registrations only fail at runtime. The fix is per-module key-path extensions declared in the module that owns the service — `extension DependencyValues { var auth: AuthService { ... } }` declared in `AuthInterface`, not in a centralized file.

**Composition root strategies** divide into three: top-level App composition (Pure DI, used by Apple samples, Point-Free `isowords`, most Tuist TMA projects), per-feature composition roots orchestrated at top level (each feature ships an `Assembly`/`Builder`), and hierarchical parent/child containers (Needle's model). For SwiftPM-first projects, **top-level App composition is the right default**; reach for hierarchical containers only when generational scoping (a `User` that exists only post-login) is a first-class architectural concern.

The real-world case studies converge on these patterns. Uber chose hierarchical compile-time DI (Needle + RIBs) because their Rider/Driver scale demanded compile-time graph correctness. Airbnb published a 12-module-type taxonomy with strict visibility rules and Dev Apps for ephemeral single-module workspaces. Bumble migrated to Tuist (rejecting SPM as too slow at 110+ modules and Bazel as too heavy). Spotify migrated to Bazel and reduced CI feedback from 80→20 minutes. Lyft enforces module-type validators. Point-Free's `isowords` is the open-source modular SwiftUI/TCA reference. **The pattern that scales across all of them is interface/impl split with the SPM (or Tuist, or Bazel) package boundary as the architectural enforcement mechanism.**

## Apple's stance: implicit endorsement of `@Environment` plus SPM

Apple has never published explicit DI guidance, but the pattern is consistent across WWDC sessions and sample apps: `@State` for view-owned state, `@Environment` for global/scoped dependencies, `@Bindable` for derived bindings, SPM packages for module boundaries. WWDC 2023 #10149 ("Discover Observation in SwiftUI") establishes the model. WWDC 2023 #10160 ("Demystify SwiftUI performance") notes that the `Observable` protocol "can also help with dependency scoping by automatically limiting the dependencies to only that which is read." WWDC 2024 introduced `@Entry` and the ability to inject `@State` directly into the Environment. Backyard Birds, Scrumdinger, Food Truck, and Fruta all use `@Environment(MyModel.self)` with no DI framework, no view-models, and SPM packages for separation of concerns.

The implicit Apple recommendation is therefore: **use SwiftUI Environment + Observation; structure your app's modules with SPM; do not introduce a container library**. There is no Apple endorsement of any container. This is the strongest possible argument for a from-scratch DI design that piggybacks on `EnvironmentValues` semantics rather than fighting them.

## Testing, performance, and the cost of macros

Testing ergonomics rank, evidence-backed: `swift-dependencies` (best — exhaustive checking via `unimplemented` sentinels, parallel-safe via TaskLocal, scoped overrides), manual init DI (simplest, always works, tedious for deep hierarchies), Factory 2.5+ (compile-safe, explicit `.test` context, parallel under Swift Testing), `@Environment` (easy for view tests, awkward for view-model tests), Needle (robust but heavy — testing requires parallel component trees), Swinject/Resolver (works but globally mutable, parallel-unsafe without `.serialized`).

The **`unimplemented` pattern** — a function that calls `XCTFail` or `Issue.record` if invoked, plus a `testValue = .unimplemented` default — is the single best testing hygiene tool in the ecosystem. It forces every test to *explicitly* override the dependencies it actually uses, surfacing accidental dependency reach. **Replicate it in any custom DI you build.**

Snapshot testing (`pointfreeco/swift-snapshot-testing`) is an under-appreciated DI consumer: tests render a view to an image and diff against a stored reference, but `.onAppear` work, dates, UUIDs, network images, and animations all break determinism. A DI design that controls dates, UUIDs, and clocks first-class (as `swift-dependencies` does) makes snapshot testing dramatically more reliable.

Performance considerations have one dominant data point: **macros are expensive to compile**. SwiftSyntax debug builds at ~20 s, release at 4+ minutes (Point-Free measurements). One xcframework benchmark on Michael Tsai's blog jumped 257 s → 2004 s (7.76×) after introducing one macro. TCA users observed ~8× longer build times after adopting a macro-using version. Mitigation: Xcode 16.4 / Swift 6.1.1 ships **prebuilt SwiftSyntax binaries** behind `defaults write com.apple.dt.Xcode IDEPackageEnablePrebuilts YES`, with measured drops from 226 s → 45 s release builds. **A from-scratch DI design must not require macros for the basic happy path** — make macros an optional ergonomic layer that boils down to plain Swift the user could write by hand, and document the prebuilt-binary mitigation for users who do adopt the macro layer.

Runtime cost is generally negligible: container lookup by `ObjectIdentifier` dictionary is tens of nanoseconds; `KeyPath` access compiles to direct field-offset accessors; existential (`any P`) calls go through dynamic dispatch but at one indirect call per invocation. **TaskLocal lookup is *not* free for hot paths** — the runtime walks parent-task metadata — so avoid `@Dependency` accesses inside tight render loops; cache to a local `let`. ARC pitfalls cluster around three mistakes: closures captured by registration blocks (`register { Foo(self.bar) }` capturing `self`), strong cycles in container singletons holding services that hold containers, and existential heap allocation when values exceed the inline buffer. Eager registration inflates cold start; prefer lazy `static let liveValue`-style registration as the default.

## Synthesis: what a from-scratch Swift 6 DI system should be

The convergent design that emerges from this research is unambiguous in its primitives and opinionated in its trade-offs. **A minimum viable DI system in Swift 6 should:**

1. **Use `KeyPath<Values, Service>` as the key.** Define a `DependencyValues`-like struct extended per-module with computed properties. Never use strings. This single decision gives refactor safety, prevents typos at compile time, and matches Apple/Point-Free convention.

2. **Require a default value on every key.** Mirror `EnvironmentKey.defaultValue` / `DependencyKey.liveValue`. Unregistered services should not be a runtime crash possibility — they should be a type error. This is the closest you can get to compile-time graph completeness without a Dagger-style codegen step.

3. **Make `struct`s of `@Sendable` closures the canonical service type.** Witnesses over protocols. This dodges PAT issues, gives one-line mocks, is `Sendable`-trivial, composes cleanly with logging/retry wrappers, and admits multiple "instances" (live, mock, preview, unimplemented) for the same shape.

4. **Store the active values in a `@TaskLocal`.** This gives free per-task scoping, structured-concurrency-correct override inheritance, and Swift Testing parallelization safety. Every other primitive falls out of this one.

5. **Provide a `withDependencies { $0.api = .mock } operation: { ... }` API** for tests, previews, and transient overrides. Modeled on `TaskLocal.withValue`, scoped to the closure.

6. **Define three modes per service: `live`, `test`, `preview`.** `test` should be `unimplemented`-by-default — fail-on-call sentinels that surface unmocked dependencies as test failures. This trichotomy is the single best ergonomic improvement over Swinject/Factory.

7. **Establish a single Composition Root at `@main`.** Forbid cross-module god containers. Per-module key-path extensions are declared in the *interface* package that owns the service.

8. **Provide `Lazy<T>` and `Factory<T>` wrappers** as `Sendable` structs holding `@Sendable () -> T` closures. Necessary for cycle-breaking and per-request scopes.

9. **Treat macros as optional ergonomic sugar, never load-bearing.** Provide an `@Entry`-style macro for the boilerplate but ensure every manual fallback works without macros so SwiftSyntax build-time costs are opt-in.

10. **Build for the modular case from day one.** Interface packages declare keys; impl packages declare `liveValue`s; mock packages depend only on interfaces. The App target is the only thing that imports impls.

**What to borrow from each existing approach:** the `liveValue`/`testValue`/`previewValue` trichotomy and `withDependencies` API from `swift-dependencies`; the compile-time-required-property design from Factory; the hierarchical generational scoping concept from Needle; the SafeDI macro vocabulary (`@Instantiated`/`@Received`/`@Forwarded`) for an optional graph-validation tier; the interface/impl split convention from Airbnb/Tuist TMA; the `@Entry`-macro ergonomics from Apple/Xcode 16; the protocol-witness style from Point-Free.

**Open questions that must be resolved during implementation:**

The first is whether to provide a **graph-validation tier** at all. SafeDI proves it is possible via macros; Needle proves it works via codegen; Point-Free deliberately rejected it for `swift-dependencies` in favor of TaskLocal scoping. The right answer depends on team size and codebase scale — for a system meant to span small apps and large modular codebases, the answer is probably "yes, optional, opt-in via a `@Component`-style macro or SwiftPM build plugin." This is the highest-effort, highest-value extension.

The second is **how to handle generational dependencies** (an `AuthenticatedUser` that exists only post-login). `swift-dependencies` punts on this (Discussion #42). Needle handles it natively with hierarchical components. The pragmatic answer for a TaskLocal-based system is to build "scoped child containers" via nested `withDependencies` blocks plus a discipline that generational dependencies are passed *into* the scope as forwarded values, with a compile-time-checked optional accessor for code outside the scope.

The third is the **non-`Sendable` dependency problem**. Apple SDK types (`AVAudioEngine`, `CoreLocation` managers, `URLSessionDelegate`-conforming classes) are not `Sendable`. The system must offer a clear actor-isolation pattern (wrap in an actor, expose via `await`) and document it as the only path. `@unchecked Sendable` should be a documented escape hatch, not a default.

The fourth is **whether to expose `@Dependency`-style property wrappers or stick to closure-scoped resolution**. Property wrappers are ergonomically better for view-models but fight `@Observable` (must be marked `@ObservationIgnored`) and have known init-ordering pitfalls. The compromise used by `swift-dependencies` — property wrappers re-resolve on every access — works but adds TaskLocal-lookup cost. A from-scratch design should benchmark before committing.

The fifth is **how to bridge to SwiftUI's `@Environment`** without duplicating two systems. `swift-dependencies` 1.7 added Environment integration; the right design probably allows an `@Entry`-declared key in `EnvironmentValues` to *also* be a key in `DependencyValues`, with one-way propagation from the SwiftUI environment into the TaskLocal context when a view body executes. This is a non-trivial concurrency design problem.

The sixth is the **build-time/macro trade**. Adopting macros for ergonomics imposes SwiftSyntax compile-time costs on consumers. The mitigation is the prebuilt-binary plugin path; the design choice is whether to require it or leave it optional.

What this research definitively settles is that the **right shape** is now visible: key-path-keyed `Sendable` value containers with required defaults, `@TaskLocal` for scoping, witness structs for services, and the interface/impl split for modular boundaries. What it does *not* settle is which combination of compile-time validation, macro ergonomics, and SwiftUI bridging best fits a specific codebase's scale and team. Those choices belong to the implementation phase — guided, ideally, by a working prototype that exercises the full graph against a real modular sample app before any API surface is frozen.
